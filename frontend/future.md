# Mangoma: Future Development Roadmap

This document outlines the plan to evolve Mangoma from a standalone web application into the central creative component of a larger, automated music generation and distribution pipeline.

The vision is to support two primary workflows:
1.  **Live Performance:** A user can use the Mangoma interface to perform and stream generative music in real-time.
2.  **Automated Content Production:** The Mangoma interface is used as an authoring tool to create "presets" that feed an automated backend pipeline, which generates and uploads full-length music tracks.

---

## Phase 1: Backend Foundation & Preset Pipeline

This phase focuses on building the backend infrastructure to handle the presets created by the "Save Preset" feature and securing API access.

### 1.1: Secure Backend API Proxy <!-- COMPLETE -->
- **Goal:** Create a backend service that acts as a proxy for all Google GenAI API calls. **This is the highest priority for security.** <!-- COMPLETE -->
- **Technology:** Node.js with a framework like Express.js, or a serverless function.
- **Tasks:
    - Create a backend endpoint (e.g., `/api/generate`) that the frontend will call instead of calling Google directly. <!-- COMPLETE -->
    - **Move the `API_KEY` from the frontend to a secure environment variable on this new backend server.** <!-- COMPLETE -->
    - The backend will receive requests from the frontend, attach the secure API key, and forward them to the Google GenAI API. <!-- COMPLETE -->

### 1.2: Preset Storage API
- **Goal:** Create an endpoint to receive and store the JSON presets from the Mangoma UI.
- **Technology:** Node.js / Express.js.
- **Tasks:
    - Create a `POST /api/presets` endpoint.
    - The endpoint will accept a JSON body matching the structure generated by the frontend.
    - Add validation and integrate with a storage solution.

### 1.3: Preset Database
- **Goal:** Set up persistent storage for the presets.
- **Technology:
    - **Simple:** Store presets as JSON files on a file system or in a cloud bucket (S3, Google Cloud Storage).
    - **Advanced:** Use a NoSQL database like Firestore or MongoDB.
- **Tasks:
    - Configure the storage solution and connect it to the Preset Database API.

### 1.4: Headless Music Generation Agent
- **Goal:** Create a script that can generate a complete audio file from a preset without a browser.
- **Technology:** A Node.js script using the `@google/genai` SDK, running on the backend.
- **Tasks:
    - The script will accept a preset JSON object as an argument.
    - It will connect to the Google GenAI API (via the secure proxy or directly, since it's on the trusted backend) to generate audio.
    - It will stitch all received audio chunks together and write the final track to a `.wav` or `.mp3` file.

## Phase 2: Live Streaming Backend

This phase focuses on building the backend required to power the "Go Live" feature.

### 2.1: Real-time Audio Ingestion Server
- **Goal:** Create a server that can receive the live audio stream from the frontend.
- **Technology:** A Node.js server with a WebSocket library like `ws`.
- **Tasks:
    - Set up a WebSocket server to listen for connections from the Mangoma frontend.
    - It will receive the `audio/webm` chunks sent by the `MediaRecorder`.

### 2.2: FFmpeg Streaming Integration
- **Goal:** Process the incoming audio and stream it to a live streaming service.
- **Technology:** FFmpeg.
- **Tasks:
    - The WebSocket server will pipe the received audio chunks into a spawned FFmpeg process.
    - Configure FFmpeg to transcode the audio and set the output format to `flv` for streaming.
    - Push the stream to an RTMP URL.

### 2.3: RTMP Streaming to YouTube/Twitch
- **Goal:** Broadcast the final audio stream to a public platform.
- **Technology:** YouTube Live Streaming API, Twitch API.
- **Tasks:
    - Securely manage the RTMP URL and stream key on the backend (using environment variables).
    - The FFmpeg process will stream directly to the provided ingest endpoint.

---

## Phase 3: The Automated Pipeline (Orchestration)

This phase connects the components into a fully automated content creation system.

### 3.1: Orchestration Agent
- **Goal:** Create a central controller to manage the automated workflow.
- **Technology:** A Node.js service using a scheduler (`node-cron`) or a message queue.
- **Tasks:
    - The agent will run on a schedule (e.g., daily).
    - It will select a preset from the database.
    - It will invoke the **Headless Music Generation Agent** (from Phase 1.4) with the selected preset.

### 3.2: Video Production & Upload
- **Goal:** Expand the pipeline to include visuals and upload the final video.
- **Tasks:
    - Once the audio track is generated, the Orchestrator will trigger other agents to create visuals, combine audio and visuals using FFmpeg, and upload the final video to YouTube.

## Deployment & Infrastructure
- **Frontend:** Host the static web app on a service like Vercel, Netlify, or GitHub Pages.
- **Backend Services:** Deploy the Node.js services (API Proxy, WebSocket Server, Orchestrator) using platforms like Heroku, Google Cloud Run, or AWS Lambda.
- **Security:** Use environment variables on the hosting platforms to securely manage all API keys (`GEMINI_API_KEY`, `YOUTUBE_API_KEY`, etc.). **Never commit secrets to the Git repository.** <!-- COMPLETE for GEMINI_API_KEY -->
